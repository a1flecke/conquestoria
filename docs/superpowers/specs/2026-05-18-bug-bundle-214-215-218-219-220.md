## PR 1 — Fix #215: Encounter fires through fog

### Root cause

`hasMetCivilizationByCurrentEvidence()` in `src/systems/discovery-system.ts` uses `hasExploredCoord()` — which accepts both `'visible'` and `'fog'` states — for three checks:

- Line 104: target city position
- Line 107: any tile in `city.ownedTiles`
- Lines 124–126: any tile with `tile.owner === targetCivId`

The unit-visibility check at line 116 correctly requires `=== 'visible'`. The city and tile-ownership checks do not, so exploring a tile early in the game and then having an opponent later expand territory onto that fog-covered tile silently triggers "You encountered France" — even though the player never saw any French units or cities.

### Fix

Move the `viewerVis` capture (currently at line 112, after the city checks) to the top of `hasMetCivilizationByCurrentEvidence()`. Replace all three `hasExploredCoord` calls with `getVisibility(viewerVis, coord) === 'visible'`:

```typescript
// line 104 — before:
if (hasExploredCoord(state, viewerCivId, city.position)) return true;
// after:
if (getVisibility(viewerVis, city.position) === 'visible') return true;

// line 107 — before:
if (city.ownedTiles.some(coord => hasExploredCoord(state, viewerCivId, coord))) return true;
// after:
if (city.ownedTiles.some(coord => getVisibility(viewerVis, coord) === 'visible')) return true;

// lines 124–126 — before:
return targetTileKeys.some(key => {
  const coord = state.map.tiles[key]?.coord;
  return coord ? hasExploredCoord(state, viewerCivId, coord) : false;
});
// after:
return targetTileKeys.some(key => {
  const coord = state.map.tiles[key]?.coord;
  return coord ? getVisibility(viewerVis, coord) === 'visible' : false;
});
```

### Tests

- **Negative (fog):** viewer explored a tile on a previous turn; that tile is now in an opponent's `ownedTiles` but in fog → `hasMetCivilizationByCurrentEvidence` returns false → no first-contact event fires.
- **Positive (visible):** viewer has a unit with current LOS on a tile owned by an opponent → returns true → first-contact fires.
- **Positive (city in sight):** viewer can currently see an opponent's city position → returns true.
- **No regression on existing paths:** breakaway, war, and treaty checks are unaffected.

## PR 2 — Fix #214: Conquered city residents cannot work land

### Root cause

`resolveMajorCityCapture()` in `src/systems/city-capture-system.ts` (occupy path, line 170) calls `recalculateTerritory()`, which internally calls `normalizeCityWorkClaims()`. `normalizeCityWorkClaims` removes workers from tiles the city no longer owns after the territory recalculation — but it **never calls `assignCityFocus()`** to fill the now-empty worker slots. The conquered city is left with `workedTiles = []` and idle population with no automatic re-assignment.

### Fix

After `recalculateTerritory()` returns `territoryResult` in the occupy path, call `normalizeCityWorkAfterTerritoryChange(territoryResult.state, cityId)` and use its result as the final state passed to `buildCaptureResult`. This function already handles the re-assignment correctly: for automatic focus (e.g. `'balanced'`) it re-runs `assignCityFocus()`; for `'custom'` focus it only removes invalid tiles.

The same fix must be applied in the reconquer-breakaway path (line 123) and the raze path (line 203) for consistency, though only the occupy path has the immediate player-visible symptom.

### Data flow after fix

```
resolveMajorCityCapture(occupy)
  → build occupiedCity (owner = newOwnerId, population halved, workedTiles copied)
  → recalculateTerritory(nextState) → tile.owner updated, normalizeCityWorkClaims strips stale workers
  → normalizeCityWorkAfterTerritoryChange(territoryResult.state, cityId)  ← NEW
      → assignCityFocus('balanced') → workedTiles populated from valid owned tiles
  → buildCaptureResult(nextState, enrichedTerritoryResult, 'occupied', 0)
```

### Tests

- Conquer a city that had 4 population and 3 worked tiles. After capture, `workedTiles.length > 0` and all entries are in the city's `ownedTiles`. Workers ≤ halved population.
- Conquer a city where territory shrinks (adjacent large civ owns nearby tiles). After capture, workers are assigned only to tiles the new owner controls.
- Player-side: opening the captured city panel shows worked tiles producing yields, not 0 workers.

## PR 3 — Fix #220: Worker interaction clarity

### Root cause (dual)

**Problem 1 — Missing turn count in warning panel.**  
`createWorkerTaskWarningPanel()` in `src/ui/worker-task-warning-panel.ts` displays `"Farm in progress"` but receives no information about how many turns remain. The player cannot decide whether to keep working or move the unit.

**Problem 2 — Premature improvement rendering on tile.**  
The tile renderer shows the final improvement icon (e.g. the farm sprite) as soon as construction starts (`improvementTurnsLeft > 0`). The player reads the tile as "already complete" but the worker says "still building." These two signals conflict.

### Fix part 1 — Turns remaining in warning panel

Add `turnsLeft: number` to `WorkerTaskWarningPanelConfig`. Update the panel title from `"Farm in progress"` to `"Farm in progress — 2 turns remaining"` (singular/plural handled: `"1 turn remaining"` vs `"N turns remaining"`).

At the call site in `main.ts` (~line 1959–1962), the tile is already accessible via `gameState.map.tiles[hexKey(gameState.units[selectedId].workerTask.coord)]`. Pass `tile.improvementTurnsLeft` as `turnsLeft`.

### Fix part 2 — Under-construction tile indicator

In the tile renderer, when rendering an improvement with `tile.improvementTurnsLeft > 0`, overlay a construction indicator on top of the improvement icon. The indicator should be:
- A small hammer icon (`🔨`) positioned bottom-right of the tile, or
- A reduced-opacity (50%) tint on the improvement sprite, with the hammer badge.

This indicator is **removed** when `improvementTurnsLeft === 0` (improvement complete). The renderer already has access to `tile.improvementTurnsLeft`.

### Tests

- `WorkerTaskWarningPanelConfig` with `turnsLeft: 3` renders `"Farm in progress — 3 turns remaining"`.
- `turnsLeft: 1` renders `"Farm in progress — 1 turn remaining"` (singular).
- Tile with `improvement: 'farm'`, `improvementTurnsLeft: 2` → construction indicator present.
- Tile with `improvement: 'farm'`, `improvementTurnsLeft: 0` → no construction indicator.

## PR 4 — Fix #219: Building placement UX

### Root cause

When `placeBuildingInGrid()` cannot find a free grid slot, it silently leaves the building in `city.buildings` but not `city.grid`. The building appears under "Unplaced buildings" in the Grid tab (view-only text, no action). No notification fires. The player has no idea the building is waiting or how to place it.

### Fix, two parts

**Part 1 — Notification on unplaced completion.**  
In the `city:building-complete` handler in `src/core/turn-manager.ts`, after processing city production, check if the completed building is in the unplaced list (`getUnplacedBuildings(city).length > 0`). If so, emit a notification of the form: `"Marketplace completed — open the Grid tab to place it"`. This is a one-liner using the existing notification sink.

**Part 2 — Manual placement UI in Grid tab.**  
In `src/ui/city-grid.ts`, update the "Unplaced buildings" section:

1. Each unplaced building row becomes a **tappable button** (not a static label).
2. Tapping an unplaced building enters a **placement mode**: open grid slots are highlighted with a "place here" affordance.
3. Tapping a highlighted slot calls a new `placeBuilding(city, buildingId, slotIndex)` helper in `city-system.ts` that moves the building from unplaced → the chosen grid slot.
4. Tapping outside any slot or the building again cancels placement mode.

`placeBuilding(city, buildingId, slotIndex)` validates: slot is unlocked, slot is empty, building is in unplaced list. Returns updated city. No auto-placement logic changes.

### Tests

- Building completes with no free grid slot → notification contains building name and "Grid tab" instruction.
- Tapping unplaced building → grid slots highlight (DOM state change).
- Tapping a valid slot → `placeBuilding` succeeds, building no longer in `getUnplacedBuildings()`.
- Tapping an invalid/occupied slot → no change, stays in placement mode.
- Cancel → highlights removed, no state mutation.

## PR 5 — Fix #218: Dock building (fishing tech payoff)

### Root cause

The `fishing` tech unlocks the Fishing Boat unit but no building. Players expect coastal infrastructure after researching fishing and see nothing in the city build panel. Non-coastal cities also show nothing, leaving players without any actionable next step.

### Fix

**1. Add `dock` to `BUILDINGS` in `src/systems/city-system.ts`:**

```typescript
dock: {
  id: 'dock',
  name: 'Dock',
  description: 'Harbor for fishing boats. Boosts coastal city food and trade.',
  cost: 60,
  techRequired: 'fishing',
  coastalRequired: true,
  yields: { food: 2, gold: 1 },
  maintenanceCost: 1,
}
```

**2. Add `coastalRequired?: boolean` to the `Building` interface in `src/core/types.ts`.**

**3. Update `getAvailableBuildings(city, completedTechs, mapTiles: GameMap['tiles'])`:**  
Add `mapTiles` as a third parameter so it can evaluate `coastalRequired`. A city is coastal if any tile in `city.ownedTiles` has `terrain === 'water'` or `terrain === 'coast'` (use a small `isCityCoastal(city, mapTiles)` helper). Filter out buildings with `coastalRequired: true` when the city is not coastal. Update all three callers: `src/ui/city-panel.ts`, `src/ai/basic-ai.ts`, and `src/systems/planning-system.ts`.

**4. Add `PRODUCTION_ICONS['dock']` entry** (required by the existing icon-coverage regression).

**5. Add `DockSprite` in `src/renderer/sprites/buildings.tsx`** (required by the catalog coverage test, which asserts every building ID in `BUILDINGS` has an entry in `BUILDING_SPRITE_CATALOG`):


**6. Register in `BUILDING_SPRITE_CATALOG` in `src/renderer/sprites/sprite-catalog.ts`:**

```typescript
dock: DockSprite,
```

The sprite visually reads as a simpler, earlier-era structure than `HarborSprite`: a wooden pier extending into water with a small fishing boat alongside, no large warehouse building.

**Animation requirements:**

The global `cq-anim-idle` float (2 s vertical bob, `translateY 0 → -3px`) is inherited automatically from `SpriteFrame` and counts as the baseline for the whole sprite. The dock additionally needs water-surface motion to feel nautical. Use **inline SVG `<animate>`** elements — not CSS — because `svgOnly=true` strips the CSS `<style>` block (browsers block CSS animations in SVGs loaded as `<img>`), but SMIL `<animate>` works in both modes.

Two specific animations to add inside the sprite body:

1. **Wave ripple on the water surface paths** — animate the `d` attribute of each wave `<path>` between two slightly different sinusoidal shapes (e.g. peaks shifting a few pixels) with `dur="3s"`, `repeatCount="indefinite"`, `calcMode="spline"`. This gives the water a continuous gentle swell.

2. **Boat rock** — wrap the small boat group in a `<g>` with a transform origin at the waterline centre, then apply `<animateTransform type="rotate" values="0;1.5;0;-1.5;0" dur="4s" repeatCount="indefinite" calcMode="spline">` to give it a subtle pitch. Stagger the start with `begin="0.8s"` so it doesn't sync exactly with the wave period.

**7. Fire a guidance notification on `tech:completed` for `'fishing'`:**  
`"Fishing unlocked — build a Dock in your coastal cities to boost food and trade"`. Routed to the current player only.

### Dock stats rationale

- Cost 60 (same tier as Granary at 50, cheaper than Market at 80) — early-mid game building
- +2 food for coastal tiles (fishing boat network)
- +1 gold for trade routes
- 1 gold maintenance (small ongoing cost keeps it meaningful)
- Coastal-only gate prevents inland cities from building it

### Tests

- Non-coastal city + `fishing` tech → Dock absent from `getAvailableBuildings`.
- Coastal city + `fishing` tech → Dock present.
- Coastal city without `fishing` tech → Dock absent.
- `PRODUCTION_ICONS` coverage test passes (existing test, no new work needed beyond adding the icon entry).
- `tech:completed` with `'fishing'` → guidance notification fires for that player.
- `tech:completed` with any other tech → no fishing notification.
