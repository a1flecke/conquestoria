# Pike Thrust + Quadruped Gait — Design

## Context

The v2 sprite animation system in `src/assets/sprite-animations-v2.css` provides a single attack animation (`cq2-attack-swing`) for all melee units: an overhead slash that rotates the weapon ~98° around the shoulder pivot. This reads correctly for a swordsman but is wrong for a pikeman, whose weapon should thrust forward along its shaft axis, not arc.

Separately, the v2 system has no quadruped gait. The two hound units (`ScoutHoundV2Sprite`, `WarHoundV2Sprite` in `design/conquestoria-sprites/lib/units-v2.jsx`) currently use `data-kind="hound"` — already established as a body-plan value in the JSDoc on `SpriteFrameV2` — and rely on the default `cq2-walk-body` keyframe with no `cq-leg-*` class hooks. Their leg `<g>` elements are already structured as outer-translate + inner `<g>` (see `units-v2.jsx:813-816` and `:843-846`), so adding class hooks to the inner `<g>` and writing matching CSS is all that's needed in the JSX layer.

This spec covers two coordinated animation additions:

1. A pike-specific weapon thrust that overrides the default slash for pikes, gated by a new `data-kind-variant` axis.
2. A diagonal-pair leg gait for `data-kind="hound"` (treating "hound" as the quadruped body-plan value, since it's already in use), with two variants (`scout`, `war`) that distinguish the two hound units by mass and tempo.

### Source of record

`design/conquestoria-sprites/lib/*.jsx` is the system of record for sprite definitions. `src/renderer/sprites/v2/*.svg.ts` files are auto-generated by `scripts/serialize-sprites.mjs` (each starts with `// Auto-generated — do not edit manually`). All JSX edits land in the `design/` tree; the `.svg.ts` files are regenerated by re-running the serializer.

## Design taxonomy: `data-kind` vs `data-kind-variant`

Before either feature, codify the rule. `data-kind` already exists; `data-kind-variant` is new.

- **`data-kind`** is the **body-plan axis**. Values determine which fundamental walk/idle/attack animations apply. Existing values (per the JSDoc on `SpriteFrameV2`): `civilian`, `melee`, `ranged`, `naval`, `hound`, `spy`, `building`. The `hound` value already names the quadruped body plan. No new values are added in this spec. Rule of thumb: if it changes how the body itself moves, it's a new kind.
- **`data-kind-variant`** is the **specialization axis**, NEW in this spec. Same body plan, different sub-element animation (weapon, gait tempo, etc.). New values introduced here: `pike` (melee variant), `scout` and `war` (hound variants). Rule of thumb: if it only changes one sub-element animation while the body stays the same, it's a variant.

This convention is documented in two places:

1. A CSS comment block at the top of `sprite-animations-v2.css`.
2. The JSDoc on `SpriteFrameV2` in `sprite-system.jsx` is extended to list `data-kind-variant` values alongside the existing `data-kind` enumeration.

### `SpriteFrameV2` accepts a `variant` prop

`SpriteFrameV2` in `units-v2.jsx` currently accepts `{ size, children, hex, hexTint, state, kind, phase }` and emits `data-state` + `data-kind` on both the wrapper `<div>` and inner `<svg>`. This spec extends it to also accept `variant` and emit `data-kind-variant` on both elements when `variant` is supplied:

```jsx
function SpriteFrameV2({
  size = 128, children, hex = true, hexTint = '#000',
  state = 'idle', kind = 'civilian', variant, phase,
}) {
  // ...
  const variantAttr = variant ? { 'data-kind-variant': variant } : {};
  return (
    <div className="cq-sprite-wrap cq-v2" data-state={state} data-kind={kind} {...variantAttr}
         style={{ '--phase': p }}>
      <svg ... data-state={state} data-kind={kind} {...variantAttr}>
        ...
      </svg>
    </div>
  );
}
```

This keeps per-sprite call sites tidy: each sprite that needs a variant just passes `variant="..."` to `SpriteFrameV2`, and the attributes propagate to both DOM levels automatically (matching the existing `data-state`/`data-kind` pattern). No sprite call site has to know which DOM element gets the attribute.

## Feature 1: Pike thrust

### JSX change

In `units-v2.jsx`, `PikemanV2Sprite` becomes:

```jsx
<SpriteFrameV2 state={state} kind="melee" variant="pike" phase={phase}>
```

The inner `<g>` of the `.cq-weapon` group (line 612 in `units-v2.jsx`, currently `<g transform="translate(54 22) rotate(-8)">`) gets `className="cq-weapon-inner"` added. The pre-existing `// FLAG:` comment block at lines 582–583 is removed once the feature lands.

### Selector

The thrust animation is keyed off two attributes and targets the inner weapon group, not `.cq-weapon` itself:

```css
.cq-v2[data-state="attack"][data-kind-variant="pike"] .cq-weapon-inner
```

Because this targets a different element than the default `.cq-v2[data-state="attack"] .cq-weapon` rule, the two rules don't conflict via specificity — both would run if both elements existed. The default slash needs to be explicitly suppressed for pikes (see "Default slash suppression" below).

### Translation axis

The pike inner group sits at `transform="translate(54 22) rotate(-8)"`. The shaft is a `<rect x="-1" y="0" width="2" height="100">` running along the local +Y axis; the spear tip is a triangle around `y=-18` (local -Y). So the shaft's "extend forward" direction is local -Y (toward the tip).

The thrust translates **on the inner group** along that local -Y axis. We do NOT rotate. We do NOT translate `.cq-weapon` in viewBox coords (which would be world-frame, not shaft-frame). The translation lives on the inner group so that if the rest-pose rotation is later tuned, the thrust direction follows automatically.

### Implementation: target the inner group, not `.cq-weapon`

The pike's rest pose `transform="translate(54 22) rotate(-8)"` lives on the inner `<g>`, NOT on `.cq-weapon` itself. Animating `.cq-weapon` directly would either ignore the rest pose (animation runs in viewBox coords, not shaft-local) or require pre-baking the rest transform into every keyframe stop (six stops to update if the rest pose ever changes).

**Solution:** add `className="cq-weapon-inner"` to the inner `<g>` of `.cq-weapon` in `PikemanV2Sprite` (the regenerated `.svg.ts` then carries it as a serialized `class`), and animate that class. The outer `.cq-weapon` still owns the `--pivot-x`/`--pivot-y` for any future hybrid moves; the inner group owns the rest pose + thrust translation. The CSS keyframe stays simple (only `translate(0, Ypx)`) because the rest transform composes in from the SVG attribute.

### Keyframes

```css
.cq-v2[data-state="attack"][data-kind-variant="pike"] .cq-weapon-inner {
  animation: cq2-attack-pike-thrust 1.4s cubic-bezier(.3,.7,.4,1) infinite;
  animation-delay: calc(var(--phase, 0) * -1.4s);
  transform-box: fill-box;
}

@keyframes cq2-attack-pike-thrust {
  0%   { transform: translate(0, 0); }
  18%  { transform: translate(0, 2px); }
  32%  { transform: translate(0, -10px); }
  48%  { transform: translate(0, -10px); }
  72%  { transform: translate(0, -2px); }
  100% { transform: translate(0, 0); }
}
```

The existing rest `transform="translate(54 22) rotate(-8)"` SVG attribute combines with the CSS transform on the inner group via standard SVG transform composition.

### Default slash suppression

Because the `[data-kind-variant="pike"]` selector targets `.cq-weapon-inner` (a different element than `.cq-weapon`), the default `cq2-attack-swing` rule on `.cq-weapon` will still try to rotate the outer group. We need to suppress it for pikes:

```css
.cq-v2[data-state="attack"][data-kind-variant="pike"] .cq-weapon {
  animation: none;
}
```

This mirrors the existing pattern in `sprite-animations-v2.css` that neutralizes the v1 walk-trail on v2 weapons (the `.cq-v2[data-state="walk"] .cq-weapon { animation: none; }` rule in the ATTACK section).

### Easing and timing

- Duration: `1.4s` — matches `cq2-attack-body`, `cq2-attack-offhand`, `cq2-plume-attack`, `cq2-hit-spark`. Pike timing must sync with the body lunge so the thrust apex coincides with the body's strike frame (32%).
- Easing: `cubic-bezier(.3,.7,.4,1)` — same as `cq2-attack-body`. The thrust accelerates fast then settles into the hold, which is how a real polearm strike feels.

### What stays unchanged

- `cq2-attack-body` (body wind-back, lunge, hold, recover) — still applies.
- `cq2-attack-offhand` (off-hand brace) — still applies. Off-hand is on the haft; bracing inward is correct for a two-handed pike.
- `cq2-plume-attack`, `cq2-hit-spark`, `cq2-cape-attack` — still apply. The hit spark fires at 32–48%, matching the thrust apex.

## Feature 2: Hound (quadruped) gait

We reuse the existing `data-kind="hound"` value rather than introducing `data-kind="quadruped"` — `hound` is already in the JSDoc enumeration of allowed `data-kind` values and used by both ScoutHound and WarHound. Naming-wise, "hound" is narrower than the body plan it represents, but renaming would touch the JSX source, the serialized `.svg.ts` files, the JSDoc, and any future cavalry/mount work would force the same decision again. We accept the slight name/scope mismatch and document in the CSS comment that `data-kind="hound"` is the quadruped body-plan slot.

### JSX changes

In `units-v2.jsx`:

```jsx
// ScoutHoundV2Sprite
<SpriteFrameV2 state={state} kind="hound" variant="scout" phase={phase}>

// WarHoundV2Sprite
<SpriteFrameV2 state={state} kind="hound" variant="war" phase={phase}>
```

The pre-existing `/* FLAG: articulated 4-leg gait needs .cq-leg-fl/fr/bl/br + new CSS. Not done here. */` comment at `units-v2.jsx:803` is removed when this feature lands.

### Class hooks on leg groups

Each hound has four leg groups already structured as `<g transform="translate(<x> <y>)"><g><rect.../></g></g>`. The inner `<g>` of each gets a leg class. Naming follows standard convention (viewer's perspective; "front" = head end, "left" = closer to foreground in this side-on sprite):

- `cq-leg-fl` — front-left
- `cq-leg-fr` — front-right
- `cq-leg-bl` — back-left
- `cq-leg-br` — back-right

For each hound sprite, the head is at the left (per `units-v2.jsx:817–818` and `:847–848`: head ellipse + snout point to the left). The four leg translates are at `x ∈ {-9.5, 0.5, 16.5, 22.5}` for ScoutHound and `{-9, 0, 16, 23}` for WarHound. The implementation will assign the four classes such that the diagonal pairs `FL+BR` and `FR+BL` are visually paired across the front/back gap. The exact mapping is determined when editing the JSX (a sprite-by-sprite decision based on which leg sits in front-vs-back); the design is correct as long as diagonals are paired.

### Diagonal-pair gait keyframes

A natural dog trot pairs FL+BR (phase A) and FR+BL (phase B). The two pairs use two different keyframes whose start positions are opposite — equivalent to a 50% animation-delay but composes cleanly with the existing `--phase` desync offset (so a row of hounds doesn't trot in lockstep).

Pivot at the top of each leg (`transform-origin: 50% 0%`). The rotation magnitudes are multiplied by a `--quad-leg-scale` CSS variable so per-variant tuning works without duplicating keyframes:

```css
.cq-v2[data-kind="hound"] .cq-leg-fl,
.cq-v2[data-kind="hound"] .cq-leg-fr,
.cq-v2[data-kind="hound"] .cq-leg-bl,
.cq-v2[data-kind="hound"] .cq-leg-br {
  transform-box: fill-box;
  transform-origin: 50% 0%;
}

@keyframes cq2-hound-leg-a {
  0%, 100% { transform: rotate(calc(-10deg * var(--quad-leg-scale, 1))); }
  50%      { transform: rotate(calc(14deg * var(--quad-leg-scale, 1))); }
}
@keyframes cq2-hound-leg-b {
  0%, 100% { transform: rotate(calc(14deg * var(--quad-leg-scale, 1))); }
  50%      { transform: rotate(calc(-10deg * var(--quad-leg-scale, 1))); }
}

.cq-v2[data-kind="hound"][data-state="walk"] .cq-leg-fl,
.cq-v2[data-kind="hound"][data-state="walk"] .cq-leg-br {
  animation: cq2-hound-leg-a var(--quad-walk-cycle, 0.6s) cubic-bezier(.4,0,.6,1) infinite;
  animation-delay: calc(var(--phase, 0) * -1 * var(--quad-walk-cycle, 0.6s));
}
.cq-v2[data-kind="hound"][data-state="walk"] .cq-leg-fr,
.cq-v2[data-kind="hound"][data-state="walk"] .cq-leg-bl {
  animation: cq2-hound-leg-b var(--quad-walk-cycle, 0.6s) cubic-bezier(.4,0,.6,1) infinite;
  animation-delay: calc(var(--phase, 0) * -1 * var(--quad-walk-cycle, 0.6s));
}
```

Default range at `--quad-leg-scale: 1` is `-10° / +14°`. WarHound sets `--quad-leg-scale: 0.85`, scaling to roughly `-8.5° / +12°`.

CSS custom properties used inside `@keyframes` (like `--quad-leg-scale`) resolve at animation-start against the target element's computed style. For static per-variant magnitudes set on a parent selector, this works in all evergreen browsers. We're not animating the variable itself — only sampling it once per animation iteration — so no `@property` registration is needed.

### Hound body bob (overrides humanoid walk)

The humanoid `cq2-walk-body` (rotate ±2°, scaleY 0.96↔1.04, translateY -3.2px) is wrong for a dog: too much vertical, too much torso roll. Quadrupeds need a flatter bob, no torso rotation, no squash. Two bob peaks per cycle because the body lifts each time a diagonal pair transitions through neutral — quadrupeds at the same cycle length as a biped have twice the landing frequency.

```css
@keyframes cq2-hound-walk-body {
  0%, 100% { transform: translateY(0); }
  25%      { transform: translateY(var(--quad-bob, -1.5px)); }
  50%      { transform: translateY(0); }
  75%      { transform: translateY(var(--quad-bob, -1.5px)); }
}

.cq-v2[data-kind="hound"][data-state="walk"] .cq-sprite-figure {
  animation: cq2-hound-walk-body var(--quad-walk-cycle, 0.6s) ease-in-out infinite;
  animation-delay: calc(var(--phase, 0) * -1 * var(--quad-walk-cycle, 0.6s));
}
```

### Hound attack: lunge

Override `cq2-attack-body` for hounds with a forward lunge (no torso rotation, no scaleX twist). Head leads the body — no separate head animation needed since it moves with the figure.

```css
@keyframes cq2-hound-attack-body {
  0%   { transform: translate(0, 0); }
  18%  { transform: translate(-3px, 2px); }                  /* crouch back */
  32%  { transform: translate(var(--quad-lunge, 8px), 0); }  /* strike lunge */
  48%  { transform: translate(var(--quad-lunge, 8px), 0); }  /* hold */
  72%  { transform: translate(1px, 1px); }                   /* recover */
  100% { transform: translate(0, 0); }
}

.cq-v2[data-kind="hound"][data-state="attack"] .cq-sprite-figure {
  animation: cq2-hound-attack-body var(--quad-attack-cycle, 1.4s) cubic-bezier(.3,.7,.4,1) infinite;
  animation-delay: calc(var(--phase, 0) * -1 * var(--quad-attack-cycle, 1.4s));
}
```

During attack, leg swing is suppressed so the body lunge reads clearly (otherwise swinging legs visually fight the forward thrust):

```css
.cq-v2[data-kind="hound"][data-state="attack"] .cq-leg-fl,
.cq-v2[data-kind="hound"][data-state="attack"] .cq-leg-fr,
.cq-v2[data-kind="hound"][data-state="attack"] .cq-leg-bl,
.cq-v2[data-kind="hound"][data-state="attack"] .cq-leg-br {
  animation: none;
}
```

The existing `cq2-hit-spark` rule is gated only on `[data-state="attack"]` and would fire automatically, but the hound JSX currently has no `.cq-hit-spark` element. Hounds will therefore have no impact flash with this change. See "Known limitations" for the follow-up.

### Hound idle

Bipedal `cq2-breathe` is approximately fine for a quadruped (slow rise/fall reads as breathing whether you have 2 or 4 legs). But the scaleY component looks wrong on a horizontal body. Override:

```css
@keyframes cq2-hound-breathe {
  0%, 100% { transform: translateY(0); }
  45%      { transform: translateY(-0.8px); }
  70%      { transform: translateY(-0.3px); }
}

.cq-v2[data-kind="hound"][data-state="idle"] .cq-sprite-figure {
  animation: cq2-hound-breathe 3.2s cubic-bezier(.4,0,.2,1) infinite;
  animation-delay: calc(var(--phase, 0) * -3.2s);
}
```

Shadow breathing (`cq2-shadow-breathe`) and shadow walk pulse (`cq2-walk-shadow`) still apply unchanged — they're kind-agnostic.

### Variants: scout vs war

The differentiation lives in CSS variables on variant-scoped selectors, not in parallel keyframe sets. Keeps the keyframes singular and the differences explicit.

```css
/* ScoutHound: light, fast, snappy — overrides the keyframe defaults */
.cq-v2[data-kind="hound"][data-kind-variant="scout"] {
  --quad-walk-cycle: 0.5s;
  --quad-bob: -1px;
  --quad-attack-cycle: 1.2s;
  --quad-lunge: 9px;
  --quad-leg-scale: 1;
}

/* WarHound: heavy, slow, committed */
.cq-v2[data-kind="hound"][data-kind-variant="war"] {
  --quad-walk-cycle: 0.75s;
  --quad-bob: -2px;
  --quad-attack-cycle: 1.6s;
  --quad-lunge: 6px;
  --quad-leg-scale: 0.85;
}
```

The attack hold duration delta (ScoutHound shorter, WarHound longer) is implicit in `--quad-attack-cycle`: the keyframe holds from 32% to 48% (a 16% span), so a 1.2s cycle gives ~192ms of hold and a 1.6s cycle gives ~256ms. ScoutHound feels snappy; WarHound feels committed.

## Verification

This is a CSS/SVG visual change with no logic or test coverage required at the unit-test level. Verification is browser-based:

1. After JSX edits, run the serializer (see Files Touched #4). Verify the regenerated `.svg.ts` files contain `data-kind-variant="pike"` on the pikeman wrapper, `data-kind-variant="scout"` on scout_hound, `data-kind-variant="war"` on war_hound, `cq-weapon-inner` on the pikeman's inner weapon group, and the four `cq-leg-*` classes on each hound's inner leg groups.
2. Run `bash scripts/run-with-mise.sh yarn dev` and open the dev server.
3. Spawn a pikeman, a ScoutHound, and a WarHound in a test game.
4. Trigger combat for the pikeman against an enemy unit. Confirm: the pike thrusts forward along its shaft axis (tip moves up-and-out, then back), no overhead slash.
5. Move both hounds. Confirm: ScoutHound legs swing in diagonal pairs at a faster, snappier tempo; WarHound legs swing in diagonal pairs at a slower, heavier tempo with shorter rotation arcs.
6. Trigger combat for both hounds. Confirm: each lunges forward (no leg swing during attack), with ScoutHound lunging farther and faster, WarHound lunging shorter but holding the strike longer.
7. Confirm the reduced-motion media query still kills all animations (the existing `prefers-reduced-motion: reduce` block uses a universal `.cq-v2 *` selector that covers all new rules).

Sprite-catalog tests (`tests/renderer/sprites/sprite-catalog.test.ts`) should continue to pass since no catalog entries change. Run `bash scripts/run-with-mise.sh yarn build` and `bash scripts/run-with-mise.sh yarn test` to confirm zero regressions before any PR.

## Known limitations

- **No hit spark on hounds.** The existing `cq2-hit-spark` rule is gated only on `[data-state="attack"]` and would fire if the element were present, but neither `ScoutHoundV2Sprite` nor `WarHoundV2Sprite` currently includes a `<g className="cq-hit-spark">` element. Adding one is a separate visual-polish change, not covered here. The lunge animation reads fine without it.
- **Leg-class assignment is sprite-specific.** Because all four legs are drawn at the same y, front/back distinction comes from the relative x positions and the direction the head faces. This is documented in the JSX edit but is not auto-derived; if a future hound sprite is added with a different layout, the implementer assigns the classes manually.

## Out of scope

- Other polearm units beyond Pikeman (Halberdier, Spearman if added later) — they can opt into `variant="pike"` once they exist, but no other current unit needs this animation.
- Quadruped animations for non-hound units (cavalry mounts, wildlife) — same pattern can be reused, but no current unit requires it. If/when cavalry arrives, we can revisit whether `data-kind="hound"` should be renamed to something more generic.
- A hound ranged variant (e.g., a war-dog that shoots) — not anticipated.
- Adding `.cq-hit-spark` elements to the hound JSX — left for a follow-up visual-polish change.
